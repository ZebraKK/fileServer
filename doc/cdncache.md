## 1. 概述

应用于 CDN 节点的文件资源缓存与代理 HTTP 服务。

- 当前目标：单节点部署，模块化设计，便于未来拆分为多服务
- 横向扩展：需在架构设计中对比两种方案的利弊：
  - 方案 A：各节点独立运行，无状态，节点间缓存不共享
  - 方案 B：引入全局中心协调服务（负责配置下发、缓存协调等）

期望输出：
- 功能设计文档（含详细需求点）
- 服务架构（组件关系与数据流）
- 代码结构（目录组织与包划分）
- 横向扩展两种方案的利弊对比与推荐
- 测试系统设计（单元测试、集成测试、压测方案）


## 2. 功能需求

### 2.1 多域名管理
- 支持多个域名，每个域名独立配置：
  - 回源地址（源站 URL，支持多源轮询）
  - 默认缓存 TTL
  - 插件链（见 2.2）
- 配置支持热加载（无需重启服务）

### 2.2 HTTP 请求处理流水线（插件化）
- 每个域名可独立配置插件链，请求进入后按顺序执行
- 插件执行结果可短路后续流程（如限流触发则直接返回 429）
- 内置插件：
  - 限流：按域名或按客户端 IP，支持令牌桶/滑动窗口算法
  - URL 改写/路由：正则匹配请求路径，重写后再进入缓存/回源逻辑
  - 请求头/响应头处理：对指定 header 进行注入、覆盖或删除

### 2.3 缓存
- 缓存 key：由域名 + 路径（改写后）+ 其他参数（比如请求 header 或者 url 参数）构成
  - 其他参数的纳入规则采用两级配置：全局默认 + 域名可覆盖
  - URL 参数纳入 key 时需规范化（参数名排序后再构造 key，避免顺序不同产生重复缓存）
- TTL：优先遵守响应 Cache-Control，否则使用域名默认配置
- 淘汰策略：LRU + TTL 双机制
- 存储后端：磁盘持久化（通过本地文件管理库，见 2.5）；可选内存热层
- 缓存击穿防护：同一资源并发 miss 时，只触发一次回源，其他请求等待结果（singleflight 模式）

### 2.4 回源（Origin Pull）
- 触发条件：cache miss、TTL 过期、管理接口强制刷新
- 每个域名配置独立源站地址，支持多源轮询
- 可配置：回源超时时长、失败重试次数

### 2.5 本地文件管理（接口设计）
默认已有专有库负责实现，本服务只需定义并调用以下接口：
- Read(key string) (io.Reader, Metadata, error)
- Write(key string, r io.Reader, meta Metadata) error
- Delete(key string) error
- Exists(key string) bool
- Stat(key string) (FileInfo, error)
- List(prefix string) ([]string, error)  // 目录刷新规则匹配时需要枚举 key

### 2.6 缓存管理 API
- 刷新缓存：支持按精确 URL、URL 前缀、或整个域名。还支持指定目录
- 目录刷新缓存，只记录刷新规则（不立即扫描删除），后续有真正请求时再匹配运用
  - 刷新规则需持久化到磁盘（通过本地文件管理库以专有 key 存储），服务重启后依然有效
- （忽略）预热：传入 URL 列表，提前主动拉取并缓存
- 状态查询：查询指定 URL 的缓存状态（命中/未命中、剩余 TTL、文件大小）


## 3. 非功能需求

### 3.1 可观测性
- 结构化日志：每次请求记录域名、路径、是否命中缓存、响应时间、状态码
- Metrics（Prometheus 格式）：
  - 请求总数、命中率、回源次数
  - 缓存存储用量
  - 各插件触发次数（如限流拒绝数）
- 日志和 metrics 均为必须项，需在代码结构中体现

### 3.2 配置管理
- YAML 配置文件
- 支持热加载（SIGHUP 或配置文件变更触发）

### 3.3 优雅关闭
- 收到 SIGTERM 后，等待进行中的请求完成再退出

### 3.4 集成测试
- 独立的文件夹。包含测试说明文档，mock 设计，测试项等
- 产出要求：测试设计文档 + 可运行的 Go 测试代码框架（含 mock 接口实现骨架）

## 4. 技术约束
- 语言：Go
- HTTP 框架：轻量级优先（如 chi、echo、gin 之一）
- 本地文件操作：只通过 2.5 定义的接口访问，不直接操作文件系统
